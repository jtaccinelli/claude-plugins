# Visualizing Consumption

Render the overlay's item-to-item dependency graph — who consumes whom — as an on-demand interactive artifact. Every edge comes from documentation Trellis already maintains: each item's `REFERENCES.md` (generated by scanning real imports) supplies the ground truth, and each item's `CONTRACT.md` supplies the ratified contract it's checked against. Nothing is scanned or inferred fresh; this skill only reads and renders what's already recorded.

Unlike the other four skills, this one dispatches no agents — there's no classification, no contract handshake, no build. It's a read-and-render pass over existing leaves, run directly in the main context.

This is explicitly ephemeral. Nothing is written to `.claude/trellis/` or the repo — republish any time to see the current state, the same way you'd re-run a report.

---

## Pre-work

1. Confirm `.claude/trellis/MANIFEST.md` exists. If not, stop and tell the user to run `/trellis:initializing-overlay` first.
2. Determine scope from `$ARGUMENTS`: a single `<concern.layer>` to focus on, or the whole overlay if omitted.
3. Glob every item leaf in scope: `.claude/trellis/*/*/*/CONTRACT.md` (optionally narrowed to the requested node's directory) — this path depth naturally excludes node-level files (`AGENT.md`, `INVENTORY.md`, `REQUESTS.md`), which live one level up. If none are found, stop and report there's nothing to visualize yet — point at `/trellis:scoping-requests`, `/trellis:building-items`, or `/trellis:populating-overlay`.

---

## Steps

### Step 1 — Collect node metadata

For each `CONTRACT.md` found, read: the item's slug and node (concern + layer, derived from its path), `Provides.name`, and `Handshake.status` (`proposed | satisfied | countered | create`). This becomes each graph node's label, group (for coloring by concern), and visual state.

### Step 2 — Collect edges from REFERENCES.md

For the same set of items, read each item's `REFERENCES.md` — format defined in `trellis/skills/building-items/templates/REFERENCES.md` — and its `Used by <N> items:` list. Each row (`` `<concern> · <layer>` → `<item-slug>` ``) is an edge **from** that consumer item **to** this item.

- An item with no `REFERENCES.md` yet, or `Used by 0 items.`, still appears in the graph as an isolated node — never dropped. An item nobody consumes is meaningful information (newly created, or genuinely unused), not something to hide.
- Cross-check each edge against the *consumer's own* `CONTRACT.md` `Consumes` table. `REFERENCES.md` is the ground truth (derived from actually scanning imports), so an edge present in `REFERENCES.md` with no matching row in the consumer's ratified `Consumes` table is real drift — code depends on something the contract never declared, or the declared shape doesn't match what's actually imported. Keep the edge; mark it unconfirmed rather than silently treating it as clean.

### Step 3 — Build the graph model

Assemble nodes (`id`, `label`, `concern`, `layer`, `status`) and edges (`from`, `to`, `confirmed: boolean`) into a single structure ready to render.

### Step 4 — Render

Load the `dataviz` skill for graph and color-palette guidance before writing any rendering code. Build a self-contained HTML artifact:

- A node-link diagram, grouped and colored by Concern (Presentation / Logic / Data — reuse a consistent 3-way semantic palette across every render of this skill so repeat viewings read the same way). Distinguish Globals nodes visually from their layer-numbered siblings — they're consumed everywhere by definition, so let a force-directed layout crowd them at the center or pin them to a fixed ring rather than letting the simulation fight their high degree.
- Interactive: clicking a node highlights its direct consumers and providers and dims everything else; filter controls for concern, layer, and status; zoom and pan for larger graphs.
- A visible distinction between confirmed and unconfirmed edges (e.g. solid vs. dashed), with a legend explaining what "unconfirmed" means — this is the graph surfacing real documentation drift, not a rendering bug.

Publish via the Artifact tool.

### Step 5 — Report

Surface the artifact URL, plus a one-line summary: item count, edge count, and unconfirmed-edge count if any are non-zero.

---

## Output

- A published Artifact URL. Nothing written to `.claude/trellis/` or the repo.

## Rules

- Never dispatch a node agent or the coordinator — this skill is read-only against documentation that already exists.
- Never write any file under `.claude/trellis/` — a visualization is not a governance artifact.
- Never invent an edge that isn't backed by `REFERENCES.md` or a consumer's `CONTRACT.md` — an item with no recorded relationship renders as an isolated node, not a guess.
- Never drop an item leaf found in Step 1 from the render, even if it has zero edges.
- Never silently resolve a mismatch between `REFERENCES.md` and a `CONTRACT.md` `Consumes` table — render it as an unconfirmed edge instead, the same "surface drift, don't paper over it" discipline used everywhere else in this plugin (Learned Patterns conflicts, `populating-overlay` findings).
